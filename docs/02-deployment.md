# 02 — Deployment Manual

> Complete procedures for running GauthierFitness in **local**, **staging** and **production**. Covers RNCP
> competencies **C2.1.1** (deployment and test environments), **C2.1.2** (continuous integration) and **C2.2.4**
> (progressive deployment).

---

## 1. Environments

| Env            | URL                                                             | Source branch | Deployment trigger                           | Usage                  |
|----------------|-----------------------------------------------------------------|---------------|----------------------------------------------|------------------------|
| **Local**      | `http://localhost:8000` (back), `http://localhost:5173` (front) | any           | Manual (`php artisan serve` + `npm run dev`) | Development            |
| **Staging**    | `https://staging.gauthierfitness.fr`                            | `develop`     | Automatic after green CI                     | Functional testing, QA |
| **Production** | `https://gauthierfitness.fr`                                    | `main`        | Manual via GitHub Actions (gate)             | Real users             |

Both remote environments run on **separate OVH VPS** so that an incident on one doesn't spread to the other
(cf. staging ransomware incident, May 2026).

## 2. Prerequisites

### Local

- **PHP 8.3+** with `pdo`, `pdo_mysql`, `mbstring`, `bcmath`, `gd`, `zip`, `pcntl` extensions
- **Composer 2.8+**
- **Node 22+** and **npm 10+**
- **Docker Desktop** or native **MySQL 8**

### Production / Staging

- **Docker** ≥ 24 + **Docker Compose v2**
- SSH key access to the VPS
- Environment variables configured in GitHub Secrets

## 3. Local Startup

### Option A - With Docker (recommended)

No local MySQL installation is required: Docker provides a complete, reproducible environment in a handful of
commands.

```bash
# Clone the three repos side by side
git clone https://github.com/CharlesGAUTHIER1999/gauthierfitness-backend.git backend
git clone https://github.com/CharlesGAUTHIER1999/gauthierfitness-frontend.git frontend
git clone https://github.com/CharlesGAUTHIER1999/gauthierfitness-infra.git infra

# Backend
cd backend
cp .env.example .env
cp .env.docker.example .env.docker   # DB_HOST=db (Docker service name, not 127.0.0.1)
docker compose up -d --wait   # waits until MySQL + the app are actually ready (build included on first run, ~2-4 min)
docker compose exec app php artisan key:generate --show   # copy the displayed key into APP_KEY= in .env.docker
docker compose up -d --force-recreate app                 # reload the container with the new key
docker compose restart nginx                               # nginx cached the old container IP — reload it
docker compose exec app php artisan migrate --seed
docker compose exec app php artisan storage:link

# Frontend (in another terminal)
cd ../frontend
cp .env.example .env.local
npm install
npm run dev
```

Access:

- API → `http://localhost:8000/api` (health check: `http://localhost:8000/api/health`)
- Frontend → `http://localhost:5173`
- MySQL → `localhost:3308` (port mapped to avoid conflicting with a local MySQL)

### Option B - Without Docker

Requires a **MySQL 8 server already installed and running locally**. `.env.example` contains placeholder
credentials (`your_db_user` / `your_db_password`) that won't work as-is: first create a matching database and user
(or adapt `.env` to your existing credentials):

```sql
CREATE
DATABASE gauthier_fitness;
CREATE
USER 'your_db_user'@'localhost' IDENTIFIED BY 'your_db_password';
GRANT ALL PRIVILEGES ON gauthier_fitness.* TO
'your_db_user'@'localhost';
```

```bash
cd backend
cp .env.example .env
composer install
php artisan key:generate
php artisan migrate --seed
php artisan storage:link   # required so product images are served
composer dev      # runs simultaneously: artisan serve, queue:listen, pail, vite

# Frontend (in another terminal)
cd ../frontend
cp .env.example .env.local
npm install
npm run dev
```

The `composer dev` script runs in parallel:

- `php artisan serve` - HTTP server
- `php artisan queue:listen` - queue worker
- `php artisan pail` - real-time colored log streamer
- `npm run dev` (backend side, for any Blade assets)

## 5. CI/CD

### GitHub Actions workflow - `backend/.github/workflows/ci-cd.yml`

```
┌────────────┐    ┌──────────┐    ┌──────────────┐    ┌──────────────┐
│  PHPUnit   │───▶│   Pint   │───▶│  Build image │───▶│ Trigger      │
│  (tests)   │    │  (lint)  │    │     GHCR     │    │ deploy infra │
└────────────┘    └──────────┘    └──────────────┘    └──────────────┘
                                     (push only)        (develop only)
```

| Job              | When                   | What                                                                             | Failure → blocks   |
|------------------|------------------------|----------------------------------------------------------------------------------|--------------------|
| `phpunit`        | push + PR              | `php artisan test --stop-on-failure` with SQLite in-memory                       | the whole chain    |
| `lint`           | after phpunit          | `pint --test` (PSR-12 style)                                                     | the whole chain    |
| `build`          | push only              | Docker buildx multi-stage → push to GHCR with tag `short-sha` + `latest` on main | deployment         |
| `trigger-deploy` | push to `develop` only | `repository_dispatch` to `gauthierfitness-infra` → deploys staging               | staging deployment |

### Frontend pipeline

The frontend has its own (similar) workflow: Vite build + ESLint + push of the nginx image serving `dist/`.

### Staging deployment - automatic

On every push to `develop` that passes green CI:

1. A Docker image is built and pushed to GHCR with the `develop-<sha>` tag.
2. The `trigger-deploy` job sends a `repository_dispatch` to the `gauthierfitness-infra` repo.
3. The `gauthierfitness-infra` workflow SSHes into the staging VPS and runs `infra/scripts/deploy-staging.sh`
   with the image tag.

### Production Deployment - Manual with Gate

To deploy to production:

1. Merge `develop` → `main` (via a reviewed Pull Request).
2. The push to `main` builds the image tagged `latest` + the sha.
3. **Manually trigger** the `deploy-prod` workflow from the GitHub Actions tab.
4. The runner runs `infra/scripts/deploy-prod.sh` over SSH on the production VPS.

The `deploy-prod.sh` script runs the following steps:

```bash
git pull origin main           # up-to-date code
docker compose pull            # new images
php artisan down --retry=5     # maintenance mode
php artisan migrate --force    # DB migrations
docker compose up -d           # restart
sleep 8                        # wait for PHP-FPM
php artisan config:cache       # Laravel caches
php artisan route:cache
php artisan view:cache
php artisan event:cache
php artisan up                 # back online
docker image prune -f          # cleanup
curl /api/health               # health check (200 expected)
```

## 6. Nginx Reverse Proxy

### Production subdomains

| Subdomain                        | Service                               | Container                           |
|----------------------------------|---------------------------------------|-------------------------------------|
| `gauthierfitness.fr`             | React frontend (SPA served as static) | `gf_nginx` → `gf_frontend`          |
| `api.gauthierfitness.fr`         | Laravel API                           | `gf_nginx` → `gf_backend` (PHP-FPM) |
| `staging.gauthierfitness.fr`     | Staging frontend                      | Separate VPS                        |
| `api-staging.gauthierfitness.fr` | Staging API                           | Separate VPS                        |

### TLS configuration

All subdomains use **Let's Encrypt** with automatic renewal via Certbot:

```bash
sudo certbot --nginx -d gauthierfitness.fr -d www.gauthierfitness.fr
sudo certbot --nginx -d api.gauthierfitness.fr
# Auto-renewal via a systemd timer (check with: systemctl list-timers | grep certbot)
```

### Security headers

```nginx
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload";
add_header X-Frame-Options "SAMEORIGIN";
add_header X-Content-Type-Options "nosniff";
add_header Referrer-Policy "strict-origin-when-cross-origin";
add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://js.stripe.com; ...";
```

### Local dev: stale upstream IP after container recreation

Nginx (local `docker-compose.yml` and the staging/prod configs alike) proxies to the backend via a static
hostname (`fastcgi_pass app:9000;` locally, `upstream backend { server backend:9000; }` in staging/prod) with
no `resolver` directive. Nginx resolves that hostname to a Docker-assigned IP **once, at worker startup**, and
keeps using it. If the backend container is recreated (e.g. `docker compose up -d --force-recreate app`), it
gets a new internal IP, but nginx keeps proxying to the old, now-dead one — resulting in a `502 Bad Gateway`
until nginx itself is restarted (`docker compose restart nginx`). `deploy-staging.sh` and `deploy-prod.sh`
already restart nginx right after recreating the backend for this exact reason; the local Quick Start does the
same (see §3 above).

## 7. Monitoring

### Health check

- Public endpoint `GET /api/health` → `{"status": "ok", "env": "production"}`
- Probed by `deploy-prod.sh` after every deployment.
- To be hooked up to an external service (UptimeRobot, BetterStack) to alert on downtime.

### Logs

| Source          | Location                                                                    |
|-----------------|-----------------------------------------------------------------------------|
| Laravel         | `backend/storage/logs/laravel.log` (daily rotation in prod via `LOG_DAILY`) |
| Nginx           | `/var/log/nginx/access.log`, `/var/log/nginx/error.log`                     |
| Docker          | `docker compose logs -f <service>`                                          |
| Stripe webhooks | Persisted in DB (`webhook_events`) in addition to the Laravel log           |

## 8. Server Security (Hardening)

Following the staging ransomware incident of May 2026, both OVH VPS were hardened:

### SSH

```bash
# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
Port 22222
AllowUsers deployer
```

Only the GitHub Actions runner's public key and the personal dev key have access.

### UFW (firewall)

```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow 22222/tcp        # SSH
ufw allow 80/tcp           # HTTP (redirect to 443)
ufw allow 443/tcp          # HTTPS
ufw enable
```

**MySQL port 3306 is closed** to the outside - accessible only via the internal Docker network (the compose `db`
service, container `gf_db`, is exposed on the `gf_network` network, **not as a host port**).

### Backups

- Encrypted daily MySQL dump (`mysqldump | gpg --encrypt`) via cron at 03:00.
- Copy stored off the VPS on OVH object storage (different region).
- Monthly manual restore test.

## 9. Emergency Rollback Procedure

If a deployment breaks production:

```bash
# On the VPS
cd /var/www/gauthierfitness
export IMAGE_TAG=<previous-stable-sha>   # e.g.: 3e5a9f1
docker compose pull
docker compose up -d
php artisan migrate:rollback --step=1       # if the migration is the cause
php artisan up
curl https://gauthierfitness.fr/api/health  # check for 200
```

The previous sha is available in the GHCR history:
`https://github.com/CharlesGAUTHIER1999/gauthierfitness-backend/pkgs/container/gauthierfitness-backend`.

See also [04-upgrade.md § Rollback](./04-upgrade.md#6-rollback) for planned rollbacks and migration management.
